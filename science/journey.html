I Made a Robot Talk — Eventually

A weekend journey getting the SunFounder PiCar-X to move, speak, and cooperate. Errors included.

It started simply enough. A PiCar-X sitting on my desk, a Raspberry Pi for a brain, and one goal: write a Python script that makes it say "Moving forward" before it moves forward. What followed was a two-hour education in drivers, audio stacks, and the particular stubbornness of hardware.

I. The Setup

The SunFounder PiCar-X v2.0 is a well-built kit. Four rubber wheels, a steering servo up front, a DC motor driving the rear axle, ultrasonic distance sensor, grayscale line sensors at the bottom, and a camera on a two-axis pan-tilt mount. On top of it all sits the robot-hat board — a custom PCB that bridges the Raspberry Pi's GPIO to all the car's hardware.

Power comes from a custom battery pack: two 18650 lithium cells in blue shrink wrap, 7.4V, 2200mAh, with a PH2.0-5P connector. The elegant part: it charges in-place via USB-C through the PiPower board. No disassembly required.

The plan was clean: write scripts on Windows, commit to GitHub, pull on the Pi, run. A tidy workflow. What actually happened involved a lot more terminal windows.

II. The First Script

The first script was four lines:

from picarx import Picarx
import time

px = Picarx()
px.forward(30)
time.sleep(2)
px.stop()

Running it produced the first of many errors:

ModuleNotFoundError: No module named 'picarx'

Fair enough. Installing picarx from PyPI produced a version mismatch with robot_hat. Installing robot_hat separately surfaced a missing pyaudio. Each fix revealed the next problem, like peeling an onion made entirely of stack traces.

III. Making It Talk

The robot-hat board has a small square speaker wired in via a JST connector at J20. I could see it in a photo of the board — roughly the size of a 50-cent piece, with a small amplifier chip (U3) sitting right next to it. I wanted the car to announce each movement before executing it.

I just want it to say "Moving forward" before it moves forward. Is that so much to ask?

It turns out, yes — for about ninety minutes.

The Pi reported "No internal audio devices found." Then we found the hifiberry DAC overlay. Then the ALSA card number changed after a reboot. Then espeak was outputting mono audio to a stereo-only DAC. Then the I2S line in /boot/firmware/config.txt was commented out:

# /boot/firmware/config.txt — the culprit
#dtparam=i2s=on      ← this line was commented out
dtparam=audio=on     ← this was conflicting

The Audio Stack

How sound actually travels from Python to speaker

pico2wave generates a WAV file → aplay sends it to plughw:2,0 (the hifiberry DAC) → I2S protocol carries the signal → the PCM5102A chip converts digital to analog → the PAM amplifier on the robot-hat boosts the signal → the speaker on J20 makes sound.

The key unlock: running sudo bash i2samp.sh from SunFounder's robot-hat repo, then rebooting. Sometimes twice.

The final command that worked:

pico2wave -w /tmp/say.wav "Moving forward"
aplay -D plughw:2,0 /tmp/say.wav
"Moving forward."

The robot spoke. In a crisp, natural voice — pico2wave (the Pico TTS engine) sounds dramatically better than espeak. Clear, human, unambiguous.

IV. Putting It Together

With movement and speech both working, the final script combined them cleanly. A say() helper handles the two-step TTS pipeline, and the main sequence walks through four movements:

def say(text):
    print(text)
    subprocess.run(["pico2wave", "-w", "/tmp/say.wav", text])
    subprocess.run(["aplay", "-D", "plughw:2,0", "/tmp/say.wav"])

say("Moving forward")
px.forward(30)
time.sleep(2)
px.stop()

say("Turning left")
px.set_dir_servo_angle(-30)
px.forward(30)
time.sleep(2)
px.stop()

And a simple shell script makes running anything from the Pi terminal feel clean:

bash run.sh main          # full demo with voice
bash run.sh basic/turn_test
bash run.sh basic/stop    # emergency stop

The Workflow

1. Write scripts on Windows in VS Code

2. Commit and push to GitHub

3. On the Pi: git pull

4. Run: bash run.sh main

SSH via VS Code Remote extension keeps the terminal and file browser both connected to the Pi. No manual copying, no SCP, no guessing.
